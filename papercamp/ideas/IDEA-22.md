### IDEA-22: Restructure and actualize repo-root docs

The repo root currently has six markdown files — `README.md`, `AGENTS.md`, `CODE_STYLE.md`, `UX_PRINCIPLES.md`, `MAIN.md`, `CHANGELOG.md` — and `README.md` itself is stale: its "Pages" section lists `Focus` (removed years ago, per `progress.md`'s 2026-06-18 entry) and is missing `Review` and `Docs`, both of which exist as real routes today (`src/app/router.tsx`). The goal: only `README.md` stays in the root; everything else that's prose documentation (not config, not auto-generated tooling output) moves into a `docs/` folder, and `README.md` itself gets rewritten to actually match the current app.

**What moves to `docs/`:**
- `CODE_STYLE.md`, `UX_PRINCIPLES.md`, `MAIN.md` — straightforward moves. They already cross-reference each other with relative markdown links (e.g. `UX_PRINCIPLES.md` links to `[CODE_STYLE.md](CODE_STYLE.md)`), so moving them together into the same `docs/` folder keeps those links valid as-is.

**What has to stay in root, as deliberate exceptions:**
- **`AGENTS.md`** — many AI coding tools (Claude Code, Codex, etc.) auto-discover this filename specifically at the repo root; moving it to `docs/` would silently break that discovery convention for anyone opening this repo with such a tool. If a slimmer root file is wanted, the pattern would be a short root `AGENTS.md` that points into `docs/` for the deeper stuff — not a full move.
- **`CHANGELOG.md`** — auto-generated by `release-please-action` every release; npm/GitHub convention expects it at the package root, and moving it would need a `release-please-config.json` change to repoint the changelog path, for no real benefit.

**Knock-on updates this requires, easy to miss:**
- `AGENTS.md` references `CODE_STYLE.md`/`UX_PRINCIPLES.md` by bare filename (`AGENTS.md:19-20`) — needs updating to `docs/CODE_STYLE.md`/`docs/UX_PRINCIPLES.md` once they move while `AGENTS.md` itself stays put.
- `.coderabbit.yaml`'s `path_instructions` has a `path: "*.md"` glob (root-only, not recursive) carrying the "follow `AGENTS.md` and markdown conventions" instruction — moved files would silently stop matching that instruction unless the glob becomes `"**/*.md"` or a dedicated `docs/*.md` entry is added.
- Whether `MAIN.md`'s manifesto-style content (the "why this exists" pitch) gets moved into `docs/` as a standalone file, or folded into the refreshed `README.md`'s intro instead — worth deciding during the build rather than just relocating it verbatim, since `README.md`'s whole point is being the one thing a visitor reads first.

**Out of scope:** the `papercamp/` folder (`plans.md`/`ideas.md`/`decisions.md`/etc.) is a separate concept — the project's structured "memory," not prose documentation — and isn't touched by this idea. See [[IDEA-20]] for that folder's own (much larger) restructuring question.
